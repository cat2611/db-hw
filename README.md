### Задание 1

Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?

1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это реализовать?

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

`Приведите ответ в свободной форме........`

#### Ответ:
1.1.  
Тип СУБД: Классическая реляционная СУБД (например, PostgreSQL, Oracle, MS SQL Server).

Почему:
- Требуется строгая целостность данных (ACID), транзакционная поддержка, чёткая схема.
- Финансовые отчёты и прогнозы подразумевают сложные аналитические запросы с JOIN, агрегациями, оконными функциями.
- Возможность использования OLAP-расширений (например, crosstab в PostgreSQL) и материализованных представлений.

1.2.  
Для лендингов: Документоориентированная NoSQL (MongoDB, CouchDB) или ключ-значение (Redis — если временные данные).
Почему: лендинги имеют разную структуру форм, гибкую схему, высокую скорость записи.

Для CRM: Гибрид — реляционная (PostgreSQL) для контактов, сделок, истории + документная для хранения произвольных полей лидов. Но по условию задачи нужна гибкая и быстрая — значит, MongoDB или Cassandra (если требуются распределённые нагрузки).

Итог по 1.2:

Лендинги → MongoDB (гибкая схема, быстрое чтение/запись).

CRM → тоже MongoDB, но с продуманными индексами (пользователи, статусы, время создания).

1.3  
Тип СУБД: Иерархическая или документо-ориентированная с поддержкой деревьев (например, MongoDB с моделью «вложенные документы» или PostgreSQL c типом jsonb).

Почему:
- Документы/правила естественно складываются в дерево (разделы, подразделы, вложения).
- Не нужны сложные транзакции между разделами.
- Быстрый доступ по заголовкам, тегам, версиям.

1.4  
Тип СУБД: Графовая (например, Neo4j, ArangoDB (мультимодельная) или PostgreSQL с расширением pgRouting).

Почему:
- Задача быстрого поиска кратчайших/оптимальных маршрутов со связями (узлы: склады, объекты, курьеры; рёбра: расстояния, время, загруженность).
- Графовые запросы типа «найти ближайшего курьера с учётом текущих заданий» — родные для графовых СУБД.
- Требование «быстро работать со связями» — графовые СУБД дают производительность на порядок выше реляционных при глубине > 2.

### Задание 2. Транзакции
2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы транзакция завершилась успешно. Ориентируйтесь на шесть действий.

2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

Приведите ответ в свободной форме.
#### Ответ
2.1  
Действие 1. Инициация и ввод данных
Пользователь (или оператор на сайте/в приложении) вводит номер телефона, сумму пополнения и выбирает способ оплаты (банковская карта, электронные деньги, через банк). Система платежного шлюза проверяет формат номера и допустимый диапазон суммы.

Действие 2. Запрос на авторизацию платежа
Платёжный шлюз отправляет запрос в платёжную систему с параметрами: сумма, получатель (агрегатор или напрямую оператор связи), идентификатор плательщика. Платёжная система запрашивает у банка-эмитента (или у платёжного сервиса пользователя) разрешение на блокировку средств.

Действие 3. Блокировка средств и получение подтверждения
Банк проверяет достаточность средств, лимиты, отсутствие блокировок и возвращает успешный код авторизации (auth code). Средства временно блокируются на счёте пользователя, но ещё не списаны. Платёжный шлюз фиксирует, что авторизация пройдена.

Действие 4. Внутренняя обработка у оператора связи (пополнение баланса)
Платёжный шлюз отправляет запрос оператору связи (через API) о зачислении суммы на указанный номер. Оператор выполняет:
- Проверку существования номера.
- Начисление средств на лицевой счёт абонента (увеличение баланса или уменьшение долга).
- Генерацию внутреннего ID транзакции.

Действие 5. Подтверждение успеха и списание средств
Оператор возвращает платёжному шлюзу ответ «успех». Платёжный шлюз передаёт финальное подтверждение в платёжную систему, которая инициирует списание ранее заблокированных средств со счёта пользователя. Платёжная система переводит деньги расчётному счёту оператора (обычерез агрегатора или прямой договор).

Действие 6. Уведомление пользователя и завершение
Пользователь получает подтверждение (в приложении, SMS, email, push-уведомление) с суммой и остатком на счёте. Система платежного шлюза и биллинг оператора фиксируют транзакцию в логах и отчётах. Баланс телефона обновлён.

### Задание 3. SQL vs NoSQL
3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL.

3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.

Приведите ответ в свободной форме.

#### Ответ
3.1  
1. Гарантия ACID и целостность данных на уровне транзакций
SQL-системы обеспечивают атомарность, согласованность, изоляцию и долговечность. Это критически важно для финансовых, банковских, биллинговых систем, где недопустимы потеря данных или частичные обновления.
У NoSQL (особенно широко-столбцовых и документных) часто жертвуют ACID в пользу доступности и производительности (CAP-теорема).

2. Сложные запросы и JOIN между разными сущностями
SQL позволяет легко связывать несколько таблиц через JOIN, выполнять подзапросы, оконные функции (ROW_NUMBER(), LAG()), группировки с фильтрацией (HAVING).
В NoSQL такие операции либо эмулируются в приложении (многократные запросы + объединение в коде), либо поддерживаются слабо.

4. Чёткая схема и контроль целостности данных (constraints)
SQL поддерживает первичные ключи, внешние ключи (ссылочную целостность), уникальные ограничения, проверочные CHECK, дефолтные значения. Это предотвращает попадание «грязных» данных.
NoSQL обычно не проверяет схему на уровне БД (schema-on-read), что приводит к разнобою типов, отсутствию обязательных полей и проблемам при анализе.

5. Стандартизация, зрелость и независимость от вендора
SQL — это ANSI/ISO-стандарт. Приложения можно переносить между PostgreSQL, Oracle, MSSQL, MySQL с минимальными правками. Существуют десятки инструментов для миграции, бэкапов, репликации, мониторинга.
У каждой NoSQL-системы свой язык запросов (MQL у MongoDB, CQL у Cassandra, Gremlin у графовых), несовместимый между собой.

6. Развитые средства отчётности, BI и аналитики
SQL-системы легко интегрируются с Tableau, Power BI, Apache Superset, а также поддерживают материализованные представления, хранимые процедуры, триггеры. Многие имеют OLAP-расширения (например, crosstab в PostgreSQL, columnstore в MSSQL).
NoSQL плохо подходит для сложной многомерной аналитики; данные обычно приходится выгружать в отдельное хранилище (ETL → ClickHouse, Vertica) перед построением отчётов.

### Задание 4. Кластеры
Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу выделено 1000 машин.

На основе какого критерия будете выбирать тип СУБД и какая модель распределённых вычислений здесь справится лучше всего и почему?

#### Ответ:
Лучшая модель: Massively Parallel Processing (MPP) with columnar storage.
Потому что:
— каждый узел хранит свою партицию данных и обрабатывает её локально.
— добавление узлов почти линейно увеличивает пропускную способность для CPU-bound задач.
— фильтры, агрегации и проекции выполняются на каждом узле до shuffle.
— перезапуск отдельных задач, а не всей работы.

Так как в задаче указано что будет большое количество вычислений и огромное количестов данных то оптимальным выбором будет использование
Комбинации Apache Spark(для вычислений) + Clickhouse(Данные)
Подойдет следующие решение по СУБД:
Apache Spark (как фреймворк) + Parquet / Iceberg / Data Lake

Почему:
- Обработку, обагщение дейтсвительно огромного количества данных не выдержит ни один кластер СУБД. Так как СУБД не предназначенны для такой работы. По этому лучше всего использовать Spark.
-  Clickhouse будет работать быстрее на уже обработанных данных

PS: 
Если мы считаем что в задаче используется "идеальные" данные то можно ограничится только кластером Clickhouse
